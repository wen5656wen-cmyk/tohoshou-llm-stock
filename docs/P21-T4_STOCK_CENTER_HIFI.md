# P21-T4 · Stock Center Consolidation · Hi-Fi Design Proposal

> 本轮只做设计。未修改任何代码 / API / 数据库。未 commit / push / deploy。
> 日期：2026-07-21 ｜ 前置：S0–S2 ✅ · T1 ✅ · T2 ✅ · T3 ✅

## 1. Design Goals

1. **股票中心成为老板唯一的股票工作台**，Screen 迁完即删。
2. **能力不减少** —— Screen 的 GPT 排序 / Style 筛选 / 卡片浏览 / 统计卡 / 分页全部有归宿。
3. **体验更简单** —— 现状是两边加起来 **7 + 1 = 8 个入口**，收敛到 **3 个视图 + 一套筛选**。
4. **信息更集中** —— 一屏内完成浏览、筛选、排序、搜索、收藏、进详情。

## 3. Information Architecture

### 3.1 核心决策：7 视图 → 3 视图

现有 7 个视图混了**两种不同的东西**：
- **数据源不同的独立清单**（ai / holdings / wait / watch / fav）
- **同一份全市场数据的不同呈现**（all / allstk）

新 IA 按「这是不是同一份数据」重新切分：

```
股票中心
├─ ① AI 推荐        ← 收盘决策 Top10，Master–Detail（不动，P19 已冻结）
├─ ② 全市场浏览      ← 唯一的股票浏览器：筛选 + 排序 + 表格/卡片 + 分页
└─ ③ 我的           ← 自选 / 持仓 / 等待买点 / 观察名单（四个清单收进一个视图的分段控件）
```

**为什么 `holdings`/`wait`/`watch`/`fav` 合成一个视图**：它们都是「我关心的一小撮股票」，数据量都在 10 量级，四个顶级 Tab 是过度切分。收进「我的」下的分段控件，顶部占位从 7 个降到 3 个。

**为什么 `allstk` 直接删**：它与 `all` 渲染完全相同的组件、相同的参数 —— 是一个纯粹的重复入口。

**为什么 AI 推荐不动**：它是 P19-T1 冻结的 Master–Detail 设计，数据源是 ClosingDecision 而非 screener，与本轮合并无关。

### 3.2 八项要求的落位

| 要求 | 落位 |
|---|---|
| ① 股票浏览 | 视图 ② 主体 |
| ② GPT 排序 | 视图 ② 排序区首项（默认排序） |
| ③ AI 评分 | 视图 ② 排序区 + 列表列 |
| ④ 六种投资风格 | 视图 ② 筛选区 Style 下拉 |
| ⑤ 多条件筛选 | 视图 ② 筛选区（基础行 + 高级折叠） |
| ⑥ 搜索 | 顶部全局搜索框（跨视图常驻，已存在） |
| ⑦ 收藏 | 列表行 ★ 按钮（真实 watchlist）+ 视图 ③ 管理 |
| ⑧ 快速进入详情 | 行点击 → `StockDetailModal`（已存在） |

**无重复入口**：搜索只在顶部一处；收藏只在行内 ★ 一处；浏览只在视图 ② 一处。

### 4.2 各区域说明

| 区域 | 展示什么 | 为什么 |
|---|---|---|
| **Header** | 全局搜索 + 3 视图分段控件 | 搜索跨视图常驻，避免每个视图各做一个（现状 Screen 与股票中心各一个 = 重复） |
| **口径条** | 「收盘快照非实时」+ 评分计算时间 | 沿用股票中心现有 `DataAsOfNote`。**这是必须保留的诚实性设计** —— 2026-07-21 用户曾误判数据过期，加了它才澄清 |
| **统计卡** | 总数 + 5 挡评级分布，**点击即筛选** | 从 Screen 迁入。它同时承担「概览」与「快捷筛选」双职能，比纯装饰的 KPI 有用 |
| **筛选区·基础** | 评级（分段）+ 风格 / 市场 / 行业（下拉） | 高频筛选常驻。风格是 Screen 独有能力的落点 |
| **筛选区·高级（折叠）** | 主题 / 价格区间 / 仅自选 / 仅收藏 / 隐藏高风险 / 重置 | 低频筛选折叠，默认收起。**默认收起 = 老板打开页面看到的是干净的三行控件** |
| **排序 + 视图切换** | 排序下拉 + 升降序 + 结果计数 + 表格/卡片切换 | 排序与筛选分行，避免一行挤 8 个控件（Screen 现状） |
| **列表** | 表格模式（默认，密度高）/ 卡片模式（Screen 迁入） | 表格适合比较，卡片适合浏览。老板日常是「扫一遍找候选」→ 默认表格 |
| **分页** | 24/页 | 从 Screen 迁入 |

## 5. Filter Flow

### 5.1 筛选项与默认值

| 筛选 | 控件 | 默认值 | 层级 | 数据来源 | 服务端? |
|---|---|---|---|---|---|
| **评级** Rating | 分段（6 挡） | 全部 | 基础 | `recommendationV2` | ✅ `?recommendationV2=` |
| **风格** Style | 下拉（6 种） | 全部 | 基础 | `stockStyle` | ✅ `?style=` |
| **市场** Market | 下拉（Prime/Standard/Growth） | 全部 | 基础 | `market` | ✅ `?market=` |
| **行业** Sector | 下拉（JPX 33） | 全部（URL `?sector=` 可预置） | 基础 | `sector` | ✅ `?sector=` |
| **主题** Theme | 下拉（14 主题） | 全部 | 高级 | `AITheme` | ❌ **需新增 API 参数** |
| **价格** Price | 区间输入 | 空 | 高级 | `latestClose` | ❌ **需新增 API 参数** |
| **仅自选** Watchlist | 勾选 | 关 | 高级 | `isWatchlist` 字段 | ⚠️ 见下 |
| **仅收藏** Favorite | 勾选 | 关 | 高级 | `/api/watchlist` 集合 | ⚠️ 见下 |
| **隐藏高风险** | 勾选 | 关 | 高级 | `highRiskFlag` | ✅ `?highRisk=false` |

### 5.2 关键原则：筛选必须服务端执行

**现状的隐患**：Screen 加载 200 行后在前端筛选，页面却写着「全市场」。实际是「adaptiveScore 前 200 名里筛」—— 筛「グロース市场 + 价值防御风格」可能得到 0 条，而全市场其实有几十只。

**新设计**：所有筛选走服务端 `where` 条件，分页也走服务端。这样「1,432 只」这个数字才是真的。

> 「仅自选 / 仅收藏」两项数据量极小（10 量级），可用**前端过滤 + 明确标注**，或让 API 接受 symbol 列表。设计上二者皆可，实现时按简单优先。

### 5.3 折叠方式

- 高级筛选**默认收起**，标题右侧显示已激活的高级条件数：`⚙ 高级筛选 (2)`
- 任一高级条件激活时，基础行下方出现可点击移除的条件 chip：`主题: 半导体设备 ✕`
- `[重置]` 一键清空全部筛选，回到默认状态

## 6. Ranking Flow

### 6.1 排序项

| 排序键 | 字段 | 默认方向 | 服务端? |
|---|---|---|---|
| **综合评分（GPT）** ⭐ 默认 | `GPTScore.finalScore ?? adaptiveScore` | 降序 | ⚠️ 需 join（见下） |
| **AI 评分** | `adaptiveScore` | 降序 | ✅ |
| **今日涨跌** | `changePct` | 降序 | ⚠️ 字段存在性待确认 |
| **成交额** | `averageTurnover20` | 降序 | ❌ 不在 StockScore，需 join `AlphaFactor` |
| **市值** | `Stock.marketCap` | 降序 | ❌ 需 join `Stock` |
| **最新更新** | `latestDate` | 降序 | ✅ |
| （保留）机会分 | `opportunityScore` | 降序 | ✅ |
| （保留）市场百分位 | `percentileRank` | **升序**（1 = 最好） | ✅ |
| （保留）20 日涨幅 | `return20d` | 降序 | ✅ |

### 6.2 默认排序

**综合评分（GPT Final Score）降序** —— 理由：这是 Screen 独有且经过 GPT 重排的口径，比裸 `adaptiveScore` 更贴近「今天该看哪几只」。同时它与 AI 推荐视图的 Top10 口径一致（Top10 也来自 GPT 重排），两个视图不打架。

### 6.3 组合规则

- **排序单选**（不做多级排序）—— 多级排序对老板场景收益极低，复杂度高
- 排序方向可切换（↑/↓）
- **排序 × 筛选自由组合**，互不影响
- 排序在服务端执行，与分页一致

### 6.4 GPT Final Score 的实现选择（需你裁定）

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| **A** | 前端 join（沿用 Screen 现状） | 零 API 改动 | 只能在已加载页内排序 → **与服务端分页冲突**，「全市场按 GPT 排序」是假的 |
| **B** | `/api/screener` 增加 `sort=finalScore`，服务端 join `GPTScore` | 排序真实、与分页一致 | 需改 API（增查询，不改语义） |

**我推荐 B。** A 会让默认排序变成一句谎话。

## 9. Risks

| # | 风险 | 类型 | 缓解 |
|---|---|---|---|
| **R1** | 默认排序改为 GPT finalScore，与老板现有肌肉记忆（adaptiveScore）不同 | 迁移 | 排序项显式命名「综合评分(GPT)」与「AI 评分」并列，一眼可切回 |
| **R2** | 若走方案 A（前端 join），「全市场按 GPT 排序」实为「前 200 名内排序」 | **正确性** | 采用方案 B，服务端排序 |
| **R3** | 前端筛选 + 服务端分页混用会出现「本页筛完只剩 3 条」的错觉 | **正确性** | 全部筛选服务端化；无法服务端化的（仅自选/仅收藏）明确标注作用域 |
| **R4** | Theme 筛选需 join `AITheme`（一股多主题，`@@unique([symbol, theme])`） | 实现 | M1 用 `symbol IN (SELECT symbol FROM ai_themes WHERE theme=?)` 子查询 |
| **R5** | 服务端分页改造触及 `/api/screener`，而该 API 被 5 处消费（股票中心、Screen、StockSearch、行业分析跳转、MarketBrowseView） | **兼容** | 新参数全部可选，不传即维持现状；M1 后逐个切换 |
| **R6** | 一次加载 3018 只做全市场统计卡，可能拖慢 | 性能 | 统计卡走独立聚合查询（`groupBy recommendationV2`），不拉全表 |
| **R7** | 卡片模式在 3018 条下的渲染压力 | 性能 | 分页 24/页天然限制；不引入虚拟滚动 |
| **R8** | 删除 Screen 后旧深链 `/screener?tab=screen` 失效 | 兼容 | 重定向到 `/decision-v2?tab=picks&view=all`，不留 404 |
| **R9** | 「我的」四清单中 `wait`/`watch` 依赖 `/api/admin/decision-overview` —— 该端点在 T6 会被迁移 | **顺序依赖** | T4 实现时保持调用不变；T6 迁移时一并更新（已登记） |
| **R10** | 股票中心是 P19 IA Freeze 页面，本轮改动幅度大 | 治理 | 本设计经你批准即构成对 Freeze 的正式变更授权 |

## 10. Final Recommendation

### 需要你裁定的 3 件事

1. **GPT 排序走 A 还是 B？** 我推荐 **B（服务端 join）** —— A 会让默认排序名不副实。B 需要 M1 改 `/api/screener`。
2. **Theme 与 Price 筛选是否本轮纳入？** 两者都需要 API 新增参数。若要压缩范围，可**先不做**，筛选区少两项，其余不受影响。
3. **视图从 7 收敛到 3 是否批准？** 特别是把 `holdings / wait / watch / fav` 四个顶级 Tab 收进「我的」—— 这改变老板现有的点击路径。

### 我的整体建议

按 **B + 纳入 Theme/Price + 7→3 收敛** 全量做。理由：这次是唯一一次合并窗口，留下半套（比如前端 join 的假排序、或散着的 7 个 Tab）会让下一轮又要返工，而返工要再动一次 P19 冻结页。

### Screen 迁移完成后还剩什么？

**什么都不剩。** 12 项能力全部有归宿，无任何独立职责。
