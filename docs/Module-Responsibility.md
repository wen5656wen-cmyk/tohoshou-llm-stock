# Module Responsibility Baseline（模块职责基线）

> **必读文档** — 任何功能开发、PR、版本发布前，均须以此为唯一标准。  
> 本文档由 P1 模块职责审计（2026-06-28）产出，随项目迭代持续维护。

---

## 总原则

**每个页面只有一个核心职责。**

一旦一个页面承担两个核心职责，该页面就需要拆分。  
一旦两个页面承担同一职责，其中一个就需要被移除或重定向。

---

## 十二个模块 · 唯一职责定义

### 1. Dashboard（今日总览）`/`

> **今天整个系统发生了什么。**

| 允许 | 禁止 |
|------|------|
| 今日推荐数量（STRONG_BUY / BUY 总计） | 回测明细表格 |

| 系统健康度（综合评分） | 持仓详情 |
| 数据新鲜度（各数据源最新日期） | 学习报告详情 |

| 系统告警 / 阻断项 | 回测数据成熟度倒计时 |
| 快捷跳转按钮（→ 各专项页面） | |

## 开发前必答四问

在新增任何功能前，必须完整回答以下问题：

```
1. 这个功能属于哪个模块？
   答：___________

2. 为什么属于这里，而不是其他模块？
   答：___________

3. 是否已经存在承担相同职责的页面？
   答：___________

4. 是否违反该模块的「禁止」列表？
   答：___________
```

如果第 4 问答案是「违反」，必须重新考虑功能归属，或提出新模块。

## PR 检查规则（Module Responsibility Check）

每个 PR 必须在 description 中包含以下检查格式：

```
## Module Responsibility Check

Module：<页面路径>
职责：<一句话描述>
新增功能：<本次变更内容>
结果：PASS / FAIL
理由：<判断依据>
```

**示例（PASS）：**

```
Module：/portfolio
职责：现在持有什么。
新增功能：持仓收益走势图
结果：PASS
理由：收益属于持仓信息范畴，符合 AI组合 职责定义。
```

**示例（FAIL）：**

```
Module：/
职责：今天整个系统发生了什么。
新增功能：三策略历史胜率表格
结果：FAIL
理由：历史胜率属于回测范畴，应放入 /backtest，Dashboard 禁止显示回测详情。
```

## 禁止事项（硬性规定）

以下行为禁止出现在任何 PR 中：

1. **一个页面承担多个核心职责**  
   例：Dashboard 同时显示「今日推荐」和「历史回测」。

2. **同一功能同时存在于两个页面**  
   例：AI信号胜率在 `/portfolio` 和 `/backtest` 都渲染。

3. **为了开发方便把组件随意放入 Dashboard**  
   Dashboard 不是调试面板。

4. **为了查看方便把回测放到 AI组合**  
   回测是历史验证，持仓是当前状态，两者职责不同。

5. **为了调试方便把系统运维数据放到用户页面**  
   `/sync`、`/admin/verify`、`/admin/mission-control` 是后台页面，不得出现在用户侧路由。

## 版本验收新增检查项

每个版本的 `CHANGELOG.md` 验收栏必须包含 Module Responsibility 检查结果：

```
Module Responsibility:
  Dashboard            PASS / FAIL
  AI选股（/screener）  PASS / FAIL
  AI组合（/portfolio） PASS / FAIL
  回测验证（/backtest）PASS / FAIL
  研究分析             PASS / FAIL
  学习报告             PASS / FAIL
  新闻资讯             PASS / FAIL
  版本中心             PASS / FAIL
  实验管理             PASS / FAIL
  控制中心             PASS / FAIL
  数据校验             PASS / FAIL
  同步状态             PASS / FAIL
```

只有所有模块 PASS，版本才可标记 `productionReady: true`。
