## 1. 文档目的（Why）

### 为什么需要模块职责基线

AI 投资系统随功能迭代，极易出现「功能堆砌」现象：  
一个开发者为了方便调试，把回测数据放进 Dashboard；  
另一个开发者为了查看方便，把 AI 信号胜率放进 AI 组合页。  
每一次都有「合理理由」，但积累之后，每个页面都承担了不属于自己的职责，  
用户不知道该去哪里找什么，维护者不知道改一个功能会影响哪些页面。

**本文档的目的，是在架构层面预防这个问题。**

通过为每个模块定义唯一职责，确保：

- 每次新增功能时，有明确的判断依据
- 每次 PR Review 时，有标准化的检查流程
- 每次版本验收时，有可量化的合规指标
- 新成员加入时，能快速理解系统信息架构

### 适用范围

本文档适用于所有涉及 **前端页面** 的开发工作：

- `app/` 目录下的所有页面
- `components/` 目录下所有可复用组件（归属模块须在组件注释中说明）
- 新增页面路由

不适用：

- 后端 API 路由（API 的职责分离另有 `docs/API_MAP.md` 规范）
- 数据库 schema 变更
- 脚本工具（`scripts/` 目录）

### 适用版本

- **生效版本**：v17.7.0（2026-06-28）起
- **审计基准**：v17.6.0 P1 模块职责归位修复后的状态

### 设计原则

**原则一：单一职责**  
每个模块只回答一个问题。当一个页面开始回答两个问题时，它就需要被拆分。

**原则二：职责不重叠**  
同一信息只出现在一个模块。如果两个页面都展示「BUY 评级数量」，其中一个必须移除。

**原则三：用户视角优先**  
模块的划分依据用户的使用场景，而非技术实现的便利性。  
「开发时数据正好在这里」不是功能归属的理由。

**原则四：重定向优于复制**  
当一个模块需要引用另一模块的数据时，使用重定向按钮，而不是复制渲染逻辑。  
Dashboard 需要让用户查看回测 → 放一个「→ 查看回测验证」按钮，而不是把回测表格搬过来。

## 2. 十二个模块·唯一职责

---

### 模块 1：Dashboard（今日总览）`/`

> **一句话：今天整个系统发生了什么。**

| 允许 | 禁止 |
|------|------|
| 今日推荐数量（STRONG_BUY / BUY 总计） | 回测明细表格 |
| 今日市场状态 / 市场温度 | 策略胜率数据 |
| 系统健康度（综合评分） | 持仓详情 |
| 数据新鲜度（各数据源最新日期） | 学习报告详情 |
| 流水线阶段运行状态（点状图） | 个股研究内容 |
| 系统告警 / 阻断项 | 回测数据成熟度倒计时 |
| 快捷跳转按钮（→ 各专项页面） | AI 信号历史胜率 |

**典型示例（PASS）：**  
今日 STRONG_BUY × 3，BUY × 31，系统健康度 88/100，数据新鲜度「每日推荐」2026-06-28。

**典型示例（FAIL）：**  
在 Dashboard 新增「近30日策略胜率 67.2%」→ 胜率属于回测范畴，违反职责。

---

### 模块 2：AI选股（今日推荐）`/screener`

> **一句话：今天应该买什么。**

| 允许 | 禁止 |
|------|------|
| 推荐股票列表 | 当前持仓 |
| AI 综合评分 / 排名 | 收益统计 |
| 推荐评级（STRONG_BUY / BUY / HOLD / …） | 历史回测结果 |
| 推荐理由 / AI 分析摘要 | 学习报告 |
| 个股跳转详情 | 系统运维数据 |

**典型示例（PASS）：**  
7374.T 综合评分 92，STRONG_BUY，推荐理由：RSI=55，技术面良好，流动性评分 87。

**典型示例（FAIL）：**  
在 AI选股 页加入「本月胜率」标签 → 胜率属于回测，应放入 `/backtest`。

---

### 模块 3：AI组合（当前持仓）`/portfolio`

> **一句话：现在持有什么。**

| 允许 | 禁止 |
|------|------|
| 当前持仓列表 | 推荐原因 |
| 每笔持仓收益 / Alpha | 回测统计 |
| 仓位占比 | AI 信号胜率（→ `/backtest`） |
| 现金余额 | 自选股管理（待独立页面） |
| 总组合 P&L | 学习报告内容 |
| 买卖记录 | 系统运维数据 |

**典型示例（PASS）：**  
持仓 7203.T，建仓价 ¥2,450，当前 ¥2,612，+6.6%，Alpha +3.2%，策略 SWING。

**典型示例（FAIL）：**  
在 AI组合 加入「AI 信号7日胜率 63%」→ 属于回测范畴，应放入 `/backtest`。

---

### 模块 4：回测验证（历史表现）`/backtest`

> **一句话：历史策略到底准不准。**

| 允许 | 禁止 |
|------|------|
| 胜率（各周期 1d/7d/30d/90d） | 当前持仓 |
| 平均收益 / Alpha | 今日推荐 |
| 最大回撤 | 新闻资讯 |
| Benchmark 对比（vs TOPIX） | 系统运维数据 |
| AI 信号统计（信号胜率 cohort） | |
| 策略分类胜率（DAY/SWING/POSITION） | |
| 回测曲线 | |

**典型示例（PASS）：**  
7d 胜率 61.3%，样本 N=48，平均收益 +2.4%，Alpha +1.8%，vs TOPIX +0.6%。

**典型示例（FAIL）：**  
在回测验证加入「今日 STRONG_BUY 列表」→ 属于 AI选股 职责。

## 3. 模块关系图

### 用户操作主流程

```
Dashboard（今日总览）
    │  今天系统发生了什么？
    │  ↓ 想知道应该买什么
AI选股（今日推荐）
    │  今天应该买什么？
    │  ↓ 买入后，追踪持仓
AI组合（当前持仓）
    │  现在持有什么？
    │  ↓ 想知道策略是否有效
回测验证（历史表现）
    │  历史策略到底准不准？
    │  ↓ 想知道为什么推荐
研究分析（因子分析）
       为什么推荐这只股票？
```

### 信息支撑链

```
新闻资讯
    │  原始新闻信息
    ↓
研究分析（新闻情绪因子输入）
    │
    ↓
AI选股（评分结果输出）
```

### 运维管理链

```
控制中心（流水线运行状态）
    │
    ↓
同步状态（数据同步进度）
    │
    ↓
数据校验（数据是否正确）
```

### 版本管理链

```
版本中心（历史版本记录）
    │
实验管理（新版本测试）
    │
学习报告（模型学习成果）
```

### 数据流向规则

| 方向 | 规则 |
|------|------|
| Dashboard → 其他页面 | 只允许单向跳转（重定向按钮），不允许 Dashboard 渲染其他模块数据 |
| 研究分析 → AI选股 | 研究分析是 AI选股 的数据来源，但页面上不互相嵌入 |
| 控制中心 → 数据校验 | 控制中心提供系统健康总览，数据校验提供具体数据问题 |
| 回测验证 ← AI信号统计 | AI信号统计（原在 AI组合）归入回测验证模块 |

## 4. 开发前必答四问

在新增任何功能前，必须完整书面回答以下四个问题：

```
问题 1：这个功能属于哪个模块？
答：___________（填写模块名称 + 路由路径）

问题 2：为什么属于这里，而不是其他模块？
答：___________（引用本文档对应模块的「允许」列表，或说明判断理由）

问题 3：是否已经存在承担相同职责的页面？
答：___________（如果存在，应归并而不是新建）

问题 4：是否违反该模块的「禁止」列表？
答：___________（如果违反，必须重新考虑功能归属）
```

**如果第 4 问答案为「违反」：**

停止开发。回到第 1 问，重新寻找正确的模块。  
如果所有模块都不适合，提出新模块申请（见第 10 节）。

## 5. PR 检查模板

每个 PR 的 description 必须包含以下格式的 Module Responsibility Check：

```
## Module Responsibility Check

Module：<页面路径>
职责：<一句话描述，来自本文档>
新增功能：<本次变更内容>
结果：PASS / FAIL
理由：<判断依据，引用本文档>
```

**示例（PASS）：**

```
## Module Responsibility Check

Module：/portfolio
职责：现在持有什么。
新增功能：持仓收益走势图（7日 P&L 曲线）
结果：PASS
理由：收益走势属于持仓信息范畴，符合 AI组合「允许：每笔持仓收益 / Alpha」条目。
```

**示例（FAIL）：**

```
## Module Responsibility Check

Module：/
职责：今天整个系统发生了什么。
新增功能：近30日三策略历史胜率表格
结果：FAIL
理由：历史胜率属于回测范畴，Dashboard 禁止显示回测详情。
      应移至 /backtest 页面。
```

**FAIL 的 PR 处理规则：**

1. PR 不得合并，必须修改
2. 开发者须更改功能归属，或重新提交「新模块申请」
3. Reviewer 须在 comment 中注明违反的具体条目

## 6. 禁止事项

以下行为被明确禁止，无论开发者给出何种理由：

**禁止 1：一个页面承担多个核心职责**  
例：Dashboard 同时显示「今日推荐」和「历史回测」。

**禁止 2：同一功能同时存在于两个页面**  
例：AI信号胜率在 `/portfolio` 和 `/backtest` 都渲染。

**禁止 3：为了开发方便把组件随意放入 Dashboard**  
Dashboard 不是调试面板，不是信息聚合器。Dashboard 只展示「今天」发生的事。

**禁止 4：为了查看方便把回测放到 AI组合**  
回测是「历史验证」，持仓是「当前状态」，两者职责不同，使用场景不同。

**禁止 5：为了调试方便把系统运维数据放到用户页面**  
`/sync`、`/admin/verify`、`/admin/mission-control` 是后台管理页面，不得在用户侧路由（`/`、`/screener`、`/portfolio`）中渲染运维数据。

**禁止 6：通过复制已有页面来实现新功能**  
禁止复制 `/backtest` 页面的回测表格，然后把副本放进 Dashboard 或 AI组合。  
必须先确认模块归属，再决定是否需要新页面。所有功能复用须通过组件 + 正确路由实现。

## 7. 页面生命周期

「页面生命周期」描述**用户在什么情况下会打开这个页面**。  
这是判断功能归属的最直接依据。

| 模块 | 使用时机 | 使用频率 | 典型用户行为 |
|------|----------|----------|-------------|
| Dashboard | 每天开盘前 | 每日 | 快速了解今天系统状态，决定是否进行操作 |
| AI选股 | 交易前 | 每日 | 查看今日推荐列表，决定买哪只股票 |
| AI组合 | 交易后 / 盘中 | 每日 | 查看当前持仓表现，决定是否调仓 |
| 回测验证 | 策略评估时 | 每周 | 评估当前策略是否有效，决定是否调整参数 |
| 研究分析 | 决策前 | 按需 | 深入分析某只股票，获取更多决策依据 |
| 学习报告 | 模型迭代时 | 每周 | 了解 AI 模型质量变化，判断是否需要重训 |
| 新闻资讯 | 市场事件后 | 按需 | 查看影响持仓股的最新新闻 |
| 版本中心 | 部署后 | 每次部署 | 确认版本状态，对比版本间模型性能 |
| 实验管理 | 测试新策略时 | 每次实验 | 监控实验进度，对比实验结果 |
| 控制中心 | 排查问题时 | 按需（偏管理员） | 查看流水线运行状态，定位数据问题 |
| 数据校验 | 部署前 / 数据异常时 | 每次部署 | 确认数据完整性，验证生产可用性 |
| 同步状态 | 同步异常时 | 按需（偏管理员） | 确认各数据源同步完成，手动触发补同步 |

**应用规则：**

- 「每日使用」的页面（Dashboard、AI选股、AI组合）应保持信息高度精炼，不放低频数据
- 「按需使用」的页面（研究分析、控制中心）可以展示更多细节
- 「管理员使用」的页面（同步状态、数据校验）不应在用户主导航中突出显示

```
  实验管理（/admin/experiments） PASS / FAIL
  控制中心（/admin/mission-control） PASS / FAIL
  数据校验（/admin/verify）      PASS / FAIL

```

## 9. 历史违规案例

### v17.6.0 修复记录（2026-06-28）

以下违规在 P1 模块职责审计中被发现，并在 v17.6.0 修复。记录于此作为反面教材，以后不得重犯。

---

**案例 1：Dashboard 展示回测摘要**

| 项目 | 内容 |
|------|------|
| 违规页面 | `/`（Dashboard） |
| 违规内容 | 「回测摘要 v2.3」— 7d/30d/90d horizon 胜率表格 |
| 根本原因 | 开发时回测 API 已实现，顺手在 Dashboard 渲染，方便查看 |
| 危害 | Dashboard 开始承担「回测验证」职责，职责边界模糊 |

| 为什么不能再犯 | Dashboard 只展示「今天」数据，历史胜率属于「过去」数据，两者时间维度不同 |

## 10. 后续扩展规则

### 新模块申请流程

当需要新增一个不属于现有12个模块的功能时，必须先走「新模块申请」流程，禁止直接新增菜单。

**申请格式：**

```
新模块申请

模块名称：___________
路由路径：___________
一句话职责：___________
为什么现有模块无法承担此职责：___________
允许展示的内容：___________
禁止展示的内容：___________
与现有模块的关系：___________
预计使用频率：___________
```

### 已规划的未来模块

以下模块尚未实现，但职责已预先定义，开发前须遵循：

---

**AI策略（待新建）**

> 一句话：策略参数如何配置。

- 允许：策略权重配置（DAY/SWING/POSITION 比例）、阈值调整、规则编辑
- 禁止：当前持仓、历史回测明细

---

**风控中心（待新建）**

> 一句话：风险在哪里。

- 允许：组合风险敞口、高风险股票预警、止损提醒
- 禁止：今日推荐、系统运维数据

---

**资金管理（待新建）**

> 一句话：资金如何分配。

- 允许：资金分配方案、买卖金额计算、仓位建议
- 禁止：历史回测、AI 评分明细

---

**推送中心（待新建）**

> 一句话：系统发出了哪些通知。

- 允许：推送历史、订阅设置、推送规则配置
- 禁止：行情数据、持仓数据

---

### 模块扩展原则

1. **先定义职责，再允许开发**  
   职责未定义的功能不得上线。

2. **不能直接新增菜单**  
   新增菜单必须对应完整的「新模块申请」记录。

3. **职责调整须更新本文档**  
   任何模块职责的调整必须同步更新 `Module-Responsibility.md`，并在 CHANGELOG 中记录。
